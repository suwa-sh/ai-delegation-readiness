# 03. 委任線引きマトリクス(検証可能性 × 正解定義可能性)

「何を AI に任せ、何を人間に残すか」の線引きそのものが設計の中心である。
本書では **検証可能性** × **正解定義可能性** の 2 軸でマトリクスを定義し、
**観察可能・再現可能** な採点質問で各軸の高/低を判定する。

## 2 軸の定義と採点質問

**各軸とも、採点質問の過半数 yes を「高」、過半数 no を「低」と判定する**。
判定の根拠は具体例とともに記録し、第三者が同じ判定に至れる粒度に揃える。

### 検証可能性(高 / 低)

判定の正誤を後から機械的に検証できるかを問う。**主観・規範依存の度合いが低いほど高**。

| 採点質問 | 答えが yes ならポイント |
|---|---|
| 第三者が同一入力で **同じ判定** を採点できるか | 1 |
| 正誤判定に必要な情報が **入力データ + 規定文書だけで揃う**か(個人の経験に依存しないか) | 1 |
| 判定結果を **機械的に再実行可能なテスト** に落とせるか(回帰テストが書けるか) | 1 |

過半数 yes(2 つ以上 yes)= 検証可能性 **高**、過半数 no = **低**。

### 正解定義可能性(高 / 低)

判定の「正解」を一意に決められるかを問う。**規定で決まる度合いが高いほど高**。

| 採点質問 | 答えが yes ならポイント |
|---|---|
| 判定の根拠となる **規定の条番号(または手続書のステップ番号)を引ける**か | 1 |
| 判定の妥当性が **ケース個別の文脈ではなく規定で決まる**か | 1 |
| **ベテランの暗黙知に頼らず**判定理由を文書化できるか | 1 |

過半数 yes(2 つ以上 yes)= 正解定義可能性 **高**、過半数 no = **低**。

## マトリクスと 3 領域

|   | **正解定義可能性 高** | **正解定義可能性 低** |
|---|---|---|
| **検証可能性 高** | 🟢 **委任 OK**(エージェントに最終判定を任せる) | 🟡 **LLM で推論補助**(エージェントが下書き、人間が承認) |
| **検証可能性 低** | 🟡 **LLM で推論補助**(エージェントが叩き台、人間が判定) | 🔴 **人間に残す**(LLM は参考意見に留める) |

- 🟢 **委任 OK**: 監査ログ最小スキーマ(`docs/02`)に従って Result を `approved`/`rejected`/
  `escalated` のいずれかで離散記録し、`uncertain` ケースは `escalated` に直結させる
- 🟡 **LLM で推論補助**: LLM が候補を出し、**人間が最終判定**する。最終判定者を Who に必ず記録
- 🔴 **人間に残す**: LLM の出力はあくまで参考。Who に LLM の関与を記録しつつ、judgement は人間に帰属

## 具体例のプロット

軸は二値(高 / 低)で判定し、3/3 yes は「強い高」、2/3 yes は「境界の高」として
**領域分類への含意**を最後の列に書く。境界の高(2/3)は委任 OK でも `escalated` の
**頻度が上がる前提**で設計する。

| 判定 | 検証可能性 | 正解定義可能性 | 領域 | 採点根拠(1 行) |
|---|---|---|---|---|
| 領収書必須項目チェック(味の素事例) | 高(3/3) | 高(3/3) | 🟢 委任 OK | 法定記載項目を機械的に照合できる |
| インボイス制度準拠チェック(味の素事例) | 高(3/3) | 高(3/3) | 🟢 委任 OK | 登録番号 + 法定要件で一意 |
| 税務上の交際費判定(味の素事例) | 高(2/3、境界) | 高(2/3、境界) | 🟢 委任 OK だが escalated 比率が高くなる | 金額閾値・人数で機械化可能、グレー時は escalate |
| 出張旅費の妥当性判定 | 高(2/3、境界) | 高(2/3、境界) | 🟢 委任 OK だが escalated 比率が高くなる | 規程の上限額と経路で判定可 |
| コーディング委任(機械的リファクタ) | 高(3/3) | 高(2/3、境界) | 🟢 委任 OK | テストが通れば検証可、規約は明文化されている |
| コーディング委任(新規アーキテクチャ設計) | 低(1/3) | 低(1/3) | 🔴 人間に残す | 「良い設計」の正解が一意でなく、文脈依存 |
| 顧客の感情に配慮した応対 | 低(0/3) | 低(0/3) | 🔴 人間に残す | 正解が文脈・個別関係性依存 |
| 法令違反の有無(明文化された法令) | 高(3/3) | 高(3/3) | 🟢 委任 OK | 条文照合は機械的、ただし最終解釈は弁護士に escalate |
| 倫理的判断(差別表現の有無など) | 高(2/3、境界) | 低(1/3) | 🟡 LLM 推論補助 | LLM が候補抽出、人間が文脈で最終判定 |
| 新規ポリシー策定 | 低(0/3) | 低(0/3) | 🔴 人間に残す | 正解が存在せず、組織判断で定義する側 |
| 採用面接の合否判定 | 低(0/3) | 低(0/3) | 🔴 人間に残す | 公平性・差別性の観点で人間最終必須 |
| 経費精算の勘定科目候補提示(最終承認は人間) | 高(2/3、境界) | 高(2/3、境界) | 🟡 LLM 推論補助(運用方針で 🟢→🟡 へ降格) | 機械的には委任 OK 領域だが、組織判断で最終を人間に残す設計を選択 |

【観測事実】味の素事例の 3 判定(領収書 / インボイス / 交際費)について、**記事から
観測できるのは「ベンダーが委任対象として選定し、規定に基づく検証可能性が公表されている」
という事実**のみ。

【設計提案】上表の象限分類は **本マトリクスへの当てはめ**(本リポでの設計解釈)であり、
記事の観測ではない。味の素事例は本マトリクスを当てると 🟢 委任 OK + 境界の高 に
分布する。他の例(コーディング委任・倫理判断・採用面接など)は完全に本リポでの一般化。
各組織は自分の業務リストに対し同じ採点質問を当てて領域に分類する。

## 委任範囲の見直しトリガー

委任 OK の判定でも、以下のイベント時には再採点が必要:

| トリガー | 想定リスク | 対応 |
|---|---|---|
| **税制改正・新会計基準** | 規定が陳腐化し、過去判定が現行規定とずれる | 規定バージョンを固定して遡及検証(`docs/02` 拡張点参照) |
| **AI モデル変更**(バージョン更新) | 同一入力に対する判定結果がリグレッションする | 過去ログから回帰テストセットを作って差分を確認 |
| **業務範囲の追加**(新しい申請種別 / 取引先) | 標準化が追いついていない領域に AI が踏み込む | 新しい業務範囲は ①〜④ 層の再点検後にのみ委任 |
| **誤承認の発見** | ④統制層の欠けが露出 | 補正フロー → ログ追記 → 採点質問の見直し |

## なぜ「採点質問の過半数」で判定するか

採点質問を 3 つに分けて「過半数 yes」で判定する設計は、以下を意図している:

- **観察可能性**:単一の主観判断ではなく、複数の具体的な質問で判定するため、第三者が
  同じ採点に至りやすい
- **再現可能性**:各質問に yes/no で答えた根拠を残せば、後から採点を検証できる
- **境界事例の検出**:3 問中 2 問だけ yes(=ぎりぎり「高」)のケースは、🟡 推論補助に
  寄せるなど慎重に判断する余地を残す

【設計提案】採点質問の数は 3 つに固定し、「過半数」で判定するルールは本リポでの設計
提案。記事は方向(検証可能性 × 正解定義可能性)を示すのみで、採点ルールは含まない。
